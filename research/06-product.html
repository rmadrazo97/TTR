Product & Tech Roadmap

Build fast, measure € recovered, fake the tax logic before you build it. POC → MVP → Platform, one country flow end-to-end before fanning out.

The one insight that shapes the build

OCR is a commodity. Reconciliation, invoice-validity, and the claim-readiness dashboard are the product. Don't out-enterprise the enterprises — out-simplify them. Win with self-serve + no-win-no-fee + "just forward the photo."

6.1 Phased roadmap

Each phase expands — the POC is the one that matters now.

POC · Concierge pilotWeeks 0–8

Goal: prove drivers grant filing rights and hand you documents, extraction is good enough, and real recoverable € can be identified and filed. Build nothing a human + no-code can fake.

Build (real): a WhatsApp/Telegram number + per-user forwarding email as the single ingest surface → LLM-vision first-pass OCR into a simple admin console / CRM. Concierge: a contracted tax expert verifies and files (or white-labels through an incumbent) while you own the customer. Stack: Twilio WhatsApp · Postmark/Mailgun inbound · LLM vision (e.g. Claude) · no-code admin console/CRM · Make/Zapier.

Success gates: ≥60% of onboarded drivers grant apoderamiento & send a doc · ≥90% field accuracy on the 4 key fields · median € recovery identified & filed / driver > 3× annual take-rate (cash lands months 4–8).

MVP · Self-serve + automated extractionMonths 2–5

First flow to automate: foreign fuel VAT (standardised Dir. 2008/9/EC portal, pure document, no GPS). For Spain the endpoint is AEAT modelo 360.

Build: camera-first PWA (no app-store gate) · dual-extractor reconcile + confidence score · Postgres data model · claim-readiness dashboard ("€X ready, €Y blocked — invoice missing VAT ID") · human-review console · success-fee billing via Stripe, charged only on recovered €.

Success gates: 100 paying carriers · auto-extract handles ≥70% no-touch · € recovered / active driver rising.

PLATFORM · Scale & automate filingMonths 6–18
  • Native mobile apps (Expo)
  • Fuel-card + TMS integrations — clean data, zero OCR
  • Tachograph .ddd + GPS ingest → per-diem day-counting
  • Multi-country rules engine (versioned config)
  • Automated e-filing — AEAT web-service / file submission (modelo 360, gasóleo census) + apoderamiento management at scale
  • White-label API for card issuers & accountants

No architecture diagram on purpose — the platform's shape will be decided by what the MVP data model proves, not committed to now.

Filing & authorization — how claims actually reach Hacienda
  • Authorization (the onboarding gate): register TTR as a colaborador social to file on clients' behalf, and/or collect per-customer apoderamientos via AEAT's Registro de Apoderamientos (REGAPO) — the customer grants it online with Cl@ve / certificate. The apoderamiento grant flow is itself a product surface to automate.
  • Submission (yes, there's an API): modelo 360 is filed electronically by presentación mediante fichero (upload a generated file) or AEAT web services (SOAP, certificate-authenticated), with a preproduction sandbox to build against.
  • Maturity: POC/MVP file via colaboración social with human sign-off → Platform auto-submits by web service and manages apoderamientos at scale.

6.2 Core technical challenges

ChallengeApproach
Multilingual messy-photo OCRDual-extractor (LLM vision + Veryfi/Mindee) + reconcile + confidence gating; human review below threshold. Bias to the lowest-hallucination model — a wrong VAT number kills a claim.
Per-country tax rulesVersioned config-driven engine (recoverability, caps, blocked categories, excise rates) — never hardcode.
Invoice validityAutomated checks: valid supplier VAT ID (VIES), required fields, gross = net + VAT, date within claim window.
E-filing & authorizationColaboración social + apoderamiento (REGAPO) for the right to file; submit modelo 360 by file upload or AEAT web service (SOAP, certificate — sandbox available). Human sign-off before every submit.
Day-counting (per-diem)Tachograph .ddd + GPS to count actual nights; store evidence for audit (German courts require nights actually in-cab).
GDPR / privacyEU-region hosting, encryption at rest, DPA with the LLM vendor (or self-host OCR for PII), consent + retention/deletion, audit log.
Fraud / accuracyConfidence thresholds, dedup (same receipt twice), anomaly flags (impossible litres/dates), mandatory human sign-off before filing.

6.3 Recommended stack & iteration principles

Stack

Supabase (Postgres + auth + storage, EU) · an LLM API (e.g. Claude) for extraction + agentic review · Veryfi/Mindee cross-check · Next PWA → Expo mobile · Twilio WhatsApp + Postmark/Mailgun inbound · Stripe (success-fee) · Temporal/job queue for the extract→validate→assemble pipeline · Cloudflare R2 for docs.

Fast-iteration principles

  1. Measure € recovered, not documents processed.
  2. No-win-no-fee from day one.
  3. Fake the tax logic (human in a simple console) before building the rules engine.
  4. OCR is a commodity; reconciliation is the product.
  5. One country flow end-to-end until € flows — then fan out.
  6. Every rejected claim feeds a rule; every human fix tunes a threshold.
